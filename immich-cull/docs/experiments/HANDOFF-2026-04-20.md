# Handoff: what we learned, what to do next

Session dates: 2026-04-18 → 2026-04-20. The goal was to improve the culling pipeline via better LLM picks. We shipped one change, measured many more, and at the end uncovered a deeper issue about how the whole problem is shaped. This doc synthesises where we landed and what to try next.

---

## 1. The headline findings, with the nuance that took us all night to earn

| metric | v1 prompt (current prod, before 2026-04-20) | v2 prompt (shipped on 2026-04-20) |
|---|---|---|
| binary user-match (picks ≥1 photo user also kept) | 77–86% | **93%** across all variants |
| F1 (precision × recall against user's actual keep set) | 51–72% | 59–70% |
| avg keeps per subgroup | 1.0–1.2 | 2.0–2.1 |

**What the numbers actually mean**
- The "+10pp binary" sounds great but hides that on 30 groups qwen_terse+v2 produces only **5/30 exact-matches** with your real keep set; another 8 are supersets (over-keep), 3 are subsets (under-keep), and **14 are partial mismatches** — v2 picked 2 but got one right and one wrong.
- Of the 30 extra photos v2 keeps vs v1, 13 are real user-keeps and 17 are photos you culled in reality. 43% hit rate on the extras. Better than the 36% base rate, far from aligned.
- The F1 lift is real but modest (~9pp for qwen, ~6pp for gemini). The remaining ~30pp gap is probably not a prompt-tuning problem — see §2.

**Gradings ≠ real decisions.** You graded many of v2's extra picks as "1 fine" or "0 perfect" because they're *acceptable* within that subgroup. But your real batch-level culling was more aggressive — singletons and other subgroup variants were better alternatives. The grader measured "is this pick acceptable in isolation?"; your workflow measures "is this pick worth keeping given the whole batch?". Those diverge, often by a lot.

## 2. The deeper issue: the problem shape may be wrong

Your instinct mid-session was right: "our whole subgroup thing is just broken." Evidence:

- User per-subgroup keep count varies 0–4 (mode=2). v2 rigidly picks 2 → rigid-count is the dominant error mode.
- The prompt contains the adaptive-count language ("keep 1 for near-identical, 2 typical, 3+ for action") and the models ignore it — they collapse to "pick 2".
- 425 of 16,142 subgroups in the queue are pseudo-bursts (screenshots grouped by clock proximity). The subgroup classifier is grouping *loosely similar* photos, not the original intent of *pure time-adjacent bursts*.
- Prompt length: 92 lines. It tells the LLM to rate stars 0–5, classify into 17 categories, group into subgroups, identify subgroup types, rank within each, pick keeps. Every instruction dilutes the others. Local models with smaller context budgets and attention capacity suffer most.
- Grading within a subgroup is an artificial lens — the real question is "does this photo earn a keep relative to everything else I shot that day".

### A simpler framing worth trying

**Pure-burst cull only.** Reduce the subgroup step to its original intent:
- A burst = same camera, same pose/subject/framing, <2 seconds between frames, high embedding similarity.
- LLM's job on a burst: pick 1 (default) or 2 (if clearly different expressions or action stages). Nothing else.
- Non-burst "similar" photos (scene variants, different angles, near-duplicates across time) are NOT bursts for LLM purposes — they ride along as singletons and the user decides in review.

**Drop the rest.** Consider:
- Remove 0–5 star rating from the prompt (unused in practice; adds noise)
- Remove categorization codes (unused downstream, long)
- Remove whole-batch subgroup discovery (too loose; covered above)
- Keep: keep/cull recommendation + 1-line note per photo

Hypothesised prompt size: 20 lines, not 92. Expected effects:
- LLM follows the surviving instructions more consistently (less to drown out)
- Smaller context = local models can take bigger batches
- Pre-filter clarifies what the LLM is responsible for
- Keep-count problem partly dissolves because we only ask "within this pure burst, 1 or 2?"

### A two-stage approach revisited

You mentioned trying two stages before but one stage didn't get images. For next time:

- **Stage A (cheap, local, programmatic)**: detect pure bursts via metadata only (time + embedding distance + device). No LLM. Bursts → stage B; everything else → singleton review.
- **Stage B (LLM, 2–5 images)**: "here are 2–5 photos from a single burst. Pick 1 or 2." Minimal prompt, explicit pure-burst framing. Images always included.

The grader you have already evaluates exactly this (pick from burst). The infrastructure is there.

## 4. Concrete things for the next session

Ordered by expected impact × effort ratio:

### 1. Prototype the simplified prompt (1–2 hours)
Write a 20-line prompt that *only* does pure-burst keep/cull. Run the `scripts/burst_discriminator_experiment.ts` machinery (already exists) with this prompt on the same 30 Stage A groups. Compare F1 vs v2's 60%. Hypothesis: F1 goes up because the model isn't juggling 5 tasks. If true, this is the new production direction.

Where to start: `scripts/burst_discriminator_experiment.ts` already has `BURST_PROMPT_TERSE` as a trimmed prompt. The structure's there.

### 2. Build a pure-burst detector (2 hours)
Programmatic, no LLM. Inputs: filename, `fileCreatedAt`, device ID, embedding (already in `Asset`). Output: burst groups where frames are <2s apart AND cosine-similar AND same device.

Then see: how many of the 16,142 current subgroups survive this stricter filter? Probably a third. Everything else becomes singletons for manual review.

Then re-run the grader experiments on *just these pure bursts*. Hypothesis: the user keep-count variance collapses — on a pure burst you almost always keep 1 (maybe 2 if eyes-closed vs eyes-open pair).

### 3. Grade at batch level, not subgroup (1 hour UI + grading work)
Extend the grader to show a whole batch, let user grade the LLM's overall keep list. Severity + over/under-keep dimensions, same scale. This replaces the weak "accept the subgroup pick?" signal with the actual "did this match your batch-level culling?" signal.

Implementation sketch:
- New endpoint `/api/batch/:id/llm-result` returns the old LLM run JSON
- Grader mode shows all N photos side-by-side with `k`/`c` markers
- One global grade per batch: F1-like score + a note
- Accumulate across enough batches (20–30) to redo variant comparison

### 4. Retry face-coverage as tie-breaker, not primary (1 hour + 30 min compute)
The first attempt failed because the prompt said "maximize coverage" and the model traded sharpness for diversity. Revised prompt: *"Priorities: sharpness > expression > composition > moment. If you're torn between two roughly equal candidates for the 2nd keeper, prefer the one whose faces differ from the first (see `faces=[...]` in the metadata). Do NOT trade sharpness for face diversity."* Re-run `scripts/face_coverage_experiment.ts`. Hypothesis: F1 > v2 baseline.

### 5. Integrate the pseudo-burst filter (30 min)
Add a new `auto_keep_patterns`-style rule for filename `Screenshot_.*\.jpg`. Optionally split per-app ("Snapchat screenshots: auto-cull; Firefox screenshots: route to review"). 1030 photos immediately addressed.

### 6. Lower-priority / exploratory
- Unsloth IQ4_XS quant of qwen3.6-35B-A3B — 2× the local throughput if it fits
- Mixed-expert routing (local first, cloud fallback on low confidence) — cheap-first strategy
- Fine-tune on state.db's 3,843 user decisions — small model beats prompted larger ones
- Peak-action and crop-duplicate detection — from user note patterns

All of these are sketched in detail in `docs/experiments/IDEAS.md`.

---

## 5. A few things NOT to do

- **Don't grade another 30 subgroup-based experiments.** The lens is wrong. See §2.
- **Don't add more instructions to the production prompt to fix a specific failure mode.** It's already bloated. Prune first.
- **Don't assume binary user-match = quality.** F1, precision, recall — and ideally batch-level overlap — are the honest metrics.
